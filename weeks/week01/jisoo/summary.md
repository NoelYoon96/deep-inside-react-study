# 1. 프론트엔드 구성 요소와 발전 과정 돌아보기
- 웹 애플리케이션의 규모가 커지면서 상태 관리의 어려움 발생 -> 컴포넌트 기반의 아키텍처가 주목 -> React 주목

## (1) 웹 개발의 종류
### 서버 사이드 개발
- 데이터베이스 또는 외부 API로 데이터를 조회하고 이를 가공하거나 변환

### 클라이언트 사이드 개발 = 프론트엔드 개발
- 유저의 입력을 처리하고 브라우저에서 실행되는 js와 같은 언어를 사용해 즉각적인 반응을 제공
- 유저 상호작용 && 인터페이스를 담당

## (2) 웹 프론트엔드의 변천사
: 프로젝트의 모듈화와 유지보수성을 향상시키기 위한 **컴포넌트 기반 개발**의 발전 과정
### DOM을 조작하는 방식의 발전 (with JS의 사용)
- DOM : HTML 문서를 트리 구조로 표현하여 JS로 요소를 동적으로 조작하는 인터페이스
  - 새로고침 X이, 실시간으로 콘텐츠 변경 가능 (ex. 텍스트, 스타일)
  - 유저 상호작용 처리 (ex. 클릭, 스크롤 등의 이벤트 탐지)
  - HTML을 트리로 만들어서, 특정 요소 구조적으로 접근 가능
  - 서버 데이터 동적으로 표시 가능

### MPA (Multi page apllication)
- 매번 다른 URL의 HTML 문서를 서버에 새로 요청하여 화면 전체를 다시 렌더링
-> 이걸 해결하기 위해 **모듈 공유** 방식이 등장

### 모듈
- 재사용 가능한 JS 파일
- BUT **모듈 의존성이 복잡** 해지는 문제 발생
-> 이걸 해결하기 위해 **번들러** 등장

### 번들러
- 원하는 라이브러리만 요청하면 의존성을 알아서 해결하는 도구 (ex. Webpack)
- 각 페이지에 여러 스크립트 태그를 삽입해야 헀던 불편 해소 가능해짐

### 패키지 매니저 (= NPM)
- npm install 명령어를 통해 node_modules 폴더에 배치
- 패키지 형태로 간편하게 관리가 가능해짐

### 템플릿 엔진
- src/ : 주요 소스 코드
- public/ : 정적 파일
- dist/ : 번들링된 최종 결과물

## (3) 명령형 vs 선언형 프로그래밍
### 명령형 프로그래밍
- 명시적으로 UI 업데이트에 대한 로직을 수행 (특정 DOM을 찾아서 작성해야함. ex. jQuery) 
- 개발자가 UI 동기화 책임 주체라는 점에서, 복잡 + 유지보수 어려움

### 선언형 프로그래밍
- 상태를 통해 무엇을 표현할지를 선언하는 방식 (= React 개발)
- 개발자는 상태 설계 + 선언 만 진행, React가 동기화를 진행해줌
